# Product Requirements Document (PRD) Template

When drafting a PRD, ensure all sections are filled out concisely. Use bullet points and tables where possible to avoid walls of text.

## 1. Executive Summary
* **Value Proposition:** [2-3 sentences explaining what this feature/product is and why we are building it.]
* **Target Audience:** [Who is this for? e.g., Shopee Sellers in Thailand.]
* **Success Metrics:** [KPIs to measure success, e.g., "Reduce time spent on market research by 50%."]

## 2. User Personas
* **Persona Name:** [e.g., Data-Driven Seller]
* **Pain Points:** [What problem are we solving for them?]
* **Goals:** [What are they trying to achieve?]

## 3. User Stories & Acceptance Criteria
*Format: As a [user], I want to [action], so that [value/benefit].*

| User Story | Acceptance Criteria (Definition of Done) | Priority (MoSCoW) |
| :--- | :--- | :--- |
| As a [User], I want to [Action], so that [Value]. | 1. [Binary condition, e.g., Can upload CSV up to 50MB] <br> 2. [Condition 2] | Must Have |
| | | |

## 4. GUI / User Flow Requirements
* **Entry Point:** [Where does the user start?]
* **Happy Path:** [The primary, frictionless flow to success.]
* **Key Interactions:** [What are the main buttons, forms, or data views?]

## 5. Non-Functional Requirements
* **Performance:** [e.g., Page load under 2 seconds.]
* **Security:** [e.g., Passwords must be hashed; API endpoints secured.]

## 6. Out of Scope
* [Explicitly list what is NOT being built in this release to prevent scope creep.]
